Dette dokument er lavet fordi projektet er vokset til et punkt, hvor det er svært at huske alle detaljer. Det er skrevet, så en person der ikke kender projektet — eller en person der har glemt det — kan sætte sig ind i det fra bunden.

🏡 Hvad er VoresHave?

Formål

VoresHave er en digital havedagbog og have-app til privat familierug. Formålet er at holde styr på alle planter i haven — hvor de er, hvad de hedder, hvordan de trives, hvornår de skal passes, og hvordan de har udviklet sig over tid.

Baggrund

Projektet startede som et simpelt registreringsværktøj og er gradvist vokset til en fuldt udstyret have-app med AI-funktioner, foto-tidslinje, vejrudsigt og brugerstyring.

Målgruppe

Privat brug — primært Steen (admin) og Linda (member). Gæster kan se haven uden at ændre noget.

Adresse

  • Live: voreshave.soenderup.dk
  • Backup-adresse: voreshave.netlify.app
  • Lokalt (dev): http://localhost:8766

Platform

Appen er en PWA (Progressive Web App) — den kan installeres på iPhone og Mac som en rigtig app. Den virker offline (begrænset) takket være en Service Worker der cacher app-skallen.

Installer på iPhone: Åbn i Safari → tryk Del-knap → "Føj til hjemmeskærm"
⚙️ Hvad kan appen?

Havestruktur

  • Områder: Forhave, Baghave osv. — øverste niveau
  • Zoner: Rosenhaven, Terrassen, Krukker osv. — kan have underzoner
  • Planter/elementer: Individuelle planter registreret i zoner

Plantregistrering

  • Navn, latinsk navn, beskrivelse, type (Træ, Busk, Staude, Etårig, Hæk, Stueplante, Krydderurt)
  • Plantedato (eller "Ukendt")
  • Trivseldata: vandbehov (3 trin), lysbehov (5 trin), jordbund (3 trin), levetid (etårig/flerårig)
  • Fotos: op til 15 stk. pr. plante med dato — sorteret som vækst-tidslinje

AI-funktioner (kræver internet)

  • Identificer plante: Tag foto → AI gætter hvad det er, returnerer navn, type og trivseldata
  • Diagnose: "Hvad fejler den?" → foto + valgfri kontekst → AI diagnosticerer symptomer og foreslår behandling
  • Anbefalinger: AI genererer sæsonspecifikke plejeråd pr. plante
  • Viden om: AI genererer fakta-tekst om planten
  • Tjek registreringer: AI-scanning af alle planter for fejlregistreringer (forkert type m.m.)

Søgning og navigation

  • Søg i plantenavne, latinnavne, beskrivelser og zonenavne
  • Svip-navigation mellem planter i samme zone
  • Zoner folder ind/ud — kun én åben ad gangen

Påmindelser og historik

  • Påmindelser med dato og valgfri tildeling til bruger
  • Historik-noter pr. plante (kan genereres automatisk af AI-diagnose)
  • Kalender-oversigt over påmindelser

Vejrudsigt

  • Kompakt vejr-widget på forsiden
  • Viser aktuel temperatur + 4-dages prognose
  • Kilde: Open-Meteo (gratis, ingen API-nøgle nødvendig)
  • Lokation sættes af admin under ☰ → Vejr & lokation

Brugerstyring

  • PIN-login (4-6 cifre) pr. bruger
  • "Husk mig" på enheden
  • 3 roller: admin, member, guest
  • Login-log: hvem har logget ind hvornår

Admin-funktioner

  • Rediger/slet zoner og planter
  • Brugerstyring (opret, rediger, slet brugere — maks 5)
  • Tjek registreringer (lokal + AI)
  • Fejl-log (se app-fejl i realtid)
  • Jordbundsmigration (genopfrisk trivseldata fra AI)

Login-flow

Appen viser en lockscreen ved opstart. Bruger vælger sit navn og taster PIN. Ved korrekt PIN logges der ind. "Husk mig"-funktion gemmer brugernavnet i localStorage så næste gang er der ingen lockscreen.

Firebase Anonymous Auth

Udover PIN-login (som er app-niveau) sker der en usynlig Firebase Anonymous Auth ved opstart. Dette er et teknisk krav for at Firestore-databasen ikke er åben for alle. Brugerne oplever ingenting — det sker automatisk i baggrunden.

Glem enhed

Hvis du er logget ind som "husket" og vil skifte bruger: ☰ → "Glem denne enhed".

🔧 Teknisk arkitektur

Teknologistack

Appen er bevidst enkelt bygget: ren HTML, CSS og JavaScript. Ingen frameworks, ingen build-trin. Hele appen er én fil: index.html.

Dataflow

Al havedata ligger i Firebase Firestore (cloud-database). Appen henter data ved opstart og lytter løbende på ændringer (realtid). Fotos gemmes i Firebase Storage. AI-kald går via Netlify Functions (serverless) som proxy til Claude API — API-nøglen er aldrig i klientkode.

Service Worker og cache

En Service Worker cacher app-skallen (HTML, ikoner) så appen virker offline. Data fra Firestore caches også automatisk af Firebase SDK.

Ved ændringer til ikoner eller andre cachede filer: bump CACHE_VERSION i sw.js. Ellers ser brugere den gamle version.

Firebase Storage

Fotos gemmes som photos/{uuid}.jpg. Bruges til zone-fotos, element-fotos og vækst-tidslinje. Maks 15 fotos pr. plante.

Firebase Authentication

Anonymous Auth er aktiveret. Alle der åbner appen får automatisk et anonymt token — kræves for at Firestore-reglerne accepterer dem. Det er usynligt for brugerne.

Sikkerhedsregler

Firestore-reglerne er sat til: "tillad læse/skrive hvis du har et Firebase Auth-token". Det betyder: ingen kan tilgå databasen uden at have åbnet appen. Reglerne deployes via Firebase CLI (ikke via Console).

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /voreshave/{document} {
      allow read, write: if request.auth != null;
    }
  }
}

Deploy sikkerhedsregler

Kræver Firebase CLI installeret og login. Kør i Terminal:

firebase deploy --only firestore:rules --project voreshave-5e7de

🤖 Claude AI (Anthropic)

Hvad er Claude?

Claude er en AI-assistent lavet af Anthropic. VoresHave bruger Claude til to ting: (1) som kodningsassistent i udviklingsprocessen via Claude Code, og (2) som AI-motor i selve appen via API-kald.

Claude i appen

Fem Netlify Functions kalder Claude API. Kaldene koster penge pr. styk (øre-beløb per kald):

  • Identificer plante: bruger Claude Sonnet (dyrere, bedre vision)
  • Diagnosticer plante: bruger Claude Sonnet
  • Anbefalinger: bruger Claude Haiku (billigere)
  • Viden om: bruger Claude Haiku
  • Tjek registreringer: bruger Claude Haiku

API-nøgle

Anthropic API-nøglen er gemt som environment variable i Netlify (ANTHROPIC_API_KEY). Den er aldrig i selve app-koden. Find den i Netlify Console under Site settings → Environment variables, eller i dit kodeordsarkiv.

Claude Code

Claude Code er det CLI-værktøj (terminal-program) der bruges til at skrive og redigere selve app-koden. Det er Steens primære udviklingsværktøj. Startes ved at åbne projektet: skriv kode i Terminal og vælg VoresHave.

⛅ Vejrudsigt (Open-Meteo)

Om Open-Meteo

Open-Meteo er en gratis vejr-API uden API-nøgle. Appen henter vejrdata direkte fra browseren (ingen server-mellemled).

Hvad der vises

  • Aktuel temperatur, nedbørschance, vindstyrke og -retning
  • 4-dages prognose med min/max temperatur

Opsætning

Admin skal sætte lokation: ☰ → "Vejr & lokation" → søg på bynavn → vælg fra liste → gem. Lokationen gemmes i Firestore (voreshave/settings) så alle brugere ser samme vejr.

Cache

Vejrdata caches i localStorage i op til 1 time. Invalideres automatisk ved lokationsskift.

💻 Udviklingsmiljø (lokalt)

Hvad er dev-miljøet?

Til udvikling køres appen lokalt på Mac. Et sæt scripts (i scripts/-mappen) styrer miljøet automatisk via Claude Code.

Start udviklingsession

  1. Åbn Terminal
  2. Skriv kode og tryk Enter
  3. Vælg VoresHave fra listen
  4. Claude Code starter — scripts kører automatisk:
    • HTTP-server startes på port 8766
    • Safari åbner og placeres i højre 33% af skærmen
    • Terminal fylder venstre 67%

Lokalt vs. live

Lokalt kan ikke teste Netlify Functions (AI-kald). AI-funktioner testes kun på det live site. Jordbunds-migration skal altid køres fra live-sitet.

Auto-reload

Safari opdateres IKKE automatisk ved filændringer. Tryk Cmd+R manuelt i Safari for at se ændringer.

Scripts (ikke deployed)

  • dev-start.sh: Startes automatisk ved session-start
  • dev-reload.sh: Kører ved filændringer (printer bare besked)
  • dev-stop.sh: Rydder op når Claude Code lukkes
🔴 Problemer der har været

Netlify function-overforbrug (maj 2026)

Sitet var nede fordi vi overskred Netlify gratis-planens 125.000 function-kald/måned. Betalt 100 kr. for ekstra kald. AI-kald (identificer, diagnose, anbefalinger) tæller som function-kald.

Mulig løsning: Migrere til Cloudflare Workers (100.000 kald/dag gratis).

Firebase test mode var åbent (maj 2026)

Firestore kørte i "test mode" — alle kunne læse og skrive til databasen uden login. Løst ved at implementere Firebase Anonymous Auth og sikkerhedsregler.

Safari AppleScript crash (maj 2026)

Dev-scripts brugte AppleScript til at styre Safari — det crashede konsekvent. Løst ved at droppe AppleScript og bruge macOS Accessibility API i stedet.

Service Worker cache-problemer

Brugere så gammel version af appen fordi SW cachede for aggressivt. Løst ved at bruge network-first strategi for HTML og bump af cache-version ved deploy.

Netlify secrets scanner false positive

Netlify blokerede deploy pga. Firebase API-nøgle i koden (false positive — Firebase web-nøgler må gerne være offentlige). Løst ved at slå secrets scanning fra i netlify.toml.

parseDrainage — inkonsistente værdier

Claude API returnerede drainage-værdier i mange formater (dansk, engelsk, forkert casing, underscore). Løst med parseDrainage()-funktion der normaliserer alle varianter.

Firestore onSnapshot race condition

Jordbundsmigration virkede ikke fordi onSnapshot overskrev de netop gemte data. Løst — migration virker nu korrekt på live-sitet.
